## 2.1 Launcher UUPS 部署事实

- Launcher 已迁移为 `ERC1967Proxy + UUPS`；canonical Launcher address 是 `IOutrunDeployer` CREATE3 部署的 ERC1967 proxy 地址，implementation 只承载逻辑。
- Launcher proxy owner 负责 UUPS 升级授权；脚本使用独立 implementation salt 与 canonical proxy salt，部署后 readiness 校验依赖 back-reference 与 fund metadata。
- 部署拓扑把 `lpTokenImplementation` 与 `preorderSettlementExecutor` 作为 first-class deployment artifacts 返回；`DeploymentResult.lpTokenImplementation` 与 `DeploymentResult.preorderSettlementExecutor` 必须参与 same-nonce 复用验证。
- Hook proxy 使用 `TransparentUpgradeableProxy(implementation, hookOwner, initializeData)`；同一 `hookOwner` 也是 Hook initializer 的 `initialOwner`，因此 Hook `owner()` 与 `ProxyAdmin.owner()` 必须一致；engine proxy 保持 `ERC1967Proxy`。same-nonce/existing Hook proxy 复用校验包括 `EXPECTED_HOOK_PROXY_CODEHASH`、admin slot、`ProxyAdmin.owner()`，并要求 `ProxyAdmin.owner() == MemeverseUniswapHook(proxy).owner()`；operational ownership transfer 必须保持两者对齐，否则复用校验拒绝 split-control 状态。
- `preorderSettlementExecutor` external helper readiness 以运行期 codehash 等于预期值（环境变量 `EXPECTED_PREORDER_SETTLEMENT_EXECUTOR_CODEHASH`）、地址非零且有代码为准，不包含 pool-manager getter 检查。
